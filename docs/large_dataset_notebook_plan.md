# Large-Dataset Notebook Scaling Plan

This branch isolates changes needed to make the example notebooks work on full
run-scale datasets such as the CARC `runs/` products. The guiding rule is that
notebooks should orchestrate work, inspect summaries, and launch dashboards; they
should not materialize multi-GB tables unless the user explicitly asks for that.

## Shared Design

- Prefer durable disk-backed artifacts: CSV for append checkpoints, Parquet or
  partitioned Parquet for downstream analysis and dashboard datasets.
- Give every compute-heavy stage a CLI path and, when useful, a Slurm submission
  helper. Notebook cells should print the command, submit when requested, and
  show how to monitor logs.
- Make resume behavior cheap. A resumed job should scan only completion keys,
  not load full result tables.
- Keep notebook previews bounded with `max_rows`, filters, or sampled/summarized
  tables.
- Push expensive table transforms into package helpers that can stream chunks or
  process partitioned datasets.

## Step 1: `step_01_ingest_and_prepare_data.ipynb`

Current risk:
- `preprocess_waveform_files(...)` can run for a long time and creates large
  metadata outputs in a notebook process.
- `build_record_coverage_table_from_trace_metadata(...)` expects the full trace
  metadata table in memory.
- Context figures are fine, but record coverage needs bounded plotting.

Plan:
- Add an `io.preprocess.slurm` helper and `svtk io preprocess-waveforms-slurm`
  command. It should call the existing preprocess CLI, write Slurm logs under
  the configured log directory, and resume from existing processed files unless
  `--overwrite` is passed.
- Add chunked record coverage construction from trace metadata:
  `build_record_coverage_table_from_trace_metadata_file(...)`, reading CSV in
  chunks or Parquet row groups and joining only required event/station origin
  fields.
- Rewrite the notebook so the default path is:
  1. write current event-station records,
  2. submit preprocessing to Slurm for large runs,
  3. load only manifest summaries and small record-coverage samples for preview.
- Keep an in-notebook local execution cell for tutorial-sized data, guarded by a
  `run_local = False` toggle.

## Step 2: `step_02_quality_control.ipynb`

Current risk:
- The notebook still builds and returns `trace_qc_summary` and `qc_summary` in
  memory, which fails on full runs. CARC produced a 530 MB trace QC table and a
  23 GB QC inventory.
- Downstream cells call `build_comparison_eligibility(qc_summary)` and plotting
  helpers with full QC tables.

Already improved:
- Slurm QC now streams trace QC checkpoints, scans metric completion keys, and
  appends metric QC rows without loading the full 23 GB inventory.

Plan:
- Rewrite the QC build cell to submit `svtk qc slurm ... --submit` by default.
- Add disk-backed QC postprocessing helpers:
  - `write_comparison_eligibility_from_qc_inventory(qc_inventory_path, output_path, chunksize=...)`
  - `build_retention_summary_from_qc_inventory(...)`
  - `build_event_station_retention_from_qc_inventory(...)`
  - `build_post_qc_record_table_from_qc_inventory(...)`
- Change figures to consume summary tables, not full `qc_summary`.
- Keep waveform preview bounded by existing `max_records` and load only the
  needed event/station rows.

## Step 3: `step_03_calculate_metrics.ipynb`

Current risk:
- The notebook uses `metric_snapshot` and `metric_figure_snapshot`, which work
  for examples but do not generalize to full runs.
- `write_metric_outputs(...)` reads the full metric table, enriches it, builds
  path tables, and prepares dashboard datasets in memory.

Existing support:
- `svtk metrics plan --manifest` can write task manifests and batch definitions.
- `svtk metrics slurm --submit` can submit a Slurm array.
- `svtk metrics merge-batches` merges batch outputs.

Plan:
- Rewrite the notebook into a disk-backed workflow:
  1. plan metric tasks from inventories and QC table,
  2. submit Slurm array,
  3. merge completed batches,
  4. run downstream output preparation as a separate chunked command.
- Add chunked output helper and CLI:
  - `svtk metrics outputs-chunked --metrics ...`
  - join event/station metadata by dictionary lookup,
  - write `metrics_long` as partitioned Parquet,
  - build `path_summary` and dashboard summaries as streaming aggregations.
- Keep diagnostic figures on a bounded subset generated by a helper such as
  `write_metric_figure_snapshot(...)` with filters and row limits.

## Step 4: `step_04_spatial_statistics.ipynb`

Current risk:
- The notebook reads the entire metric source table and normalizes it in memory.
- Spatial correlation, clustering, PCA, and geology contrasts can become very
  large if run across all metrics/components/bands at once.

Plan:
- Add a partition-aware loader:
  `iter_metric_partitions(metric_root, filters={metric, component, band, model})`.
- Add a Slurm or local chunked spatial workflow:
  `svtk spatial stats --metrics-root ... --output-dir ... --metric PGA --band ...`
  that processes one metric/component/band/model group per job or chunk.
- Persist intermediate products per group:
  `station_bias`, `residual_grid`, `morans_i`, `distance_bins`, `clusters`,
  `pca_scores`, `pca_loadings`, and `geology_contrasts`.
- Rewrite the notebook to loop over small configured groups and read already
  written per-group products for plotting.

## Step 5: `step_05_maps_and_figures.ipynb`

Current risk:
- The notebook reads all metrics and repeatedly annotates them with GeoJSON
  regions.
- Region/corridor examples should not require full metrics when a filtered
  metric/component/band subset is enough.

Plan:
- Add chunked GeoJSON annotation helper for metrics:
  `write_geojson_annotated_metrics(...)`, supporting station and event targets
  and writing partitioned Parquet.
- For figures, add filtered loaders that read only metric/component/band/model
  partitions needed by each figure.
- Keep corridor/path selection based on `event_station_records`, which is much
  smaller than metric inventories, but write selected path IDs to disk for reuse.
- Rewrite the notebook to:
  1. build or load region annotations,
  2. select bounded metric subsets,
  3. plot from persisted subsets.

## Step 6: `step_06_additional_plotting_options.ipynb`

Current risk:
- Pattern similarity and flexible plotting cells load broad metric tables.
- Waveform map previews are already bounded but should use QC-passed record
  subsets from disk.

Plan:
- Use the Step 3 partitioned metric outputs and Step 2 comparison-eligible table
  instead of full metrics.
- Add a pattern-similarity helper that can operate on one filtered partition and
  optionally write station-anomaly products to disk.
- Rewrite flexible plotting cells to use sampled/filtered dataframes with clear
  row counts and filters printed before plotting.

## Step 7: `step_07_dashboards.ipynb`

Current risk:
- `metrics = load_output_table("metrics_enriched")` and
  `qc_summary = load_output_table("qc_inventory")` attempt to load large tables.
- Dashboard datasets should be built from partitioned metrics and summaries, not
  from full in-memory dataframes.

Plan:
- Make dashboard dataset preparation fully disk-backed:
  - metrics dashboard reads partitioned `dashboard_metrics`,
  - summary parquet/csv files are generated by chunked aggregations,
  - QC dashboard points directly at `qc_inventory.csv` or a partitioned QC
    dataset, with lazy filtering in the dashboard app where possible.
- Rewrite the notebook to call CLI commands that prepare dashboard artifacts and
  then print launch commands.

## Implementation Order

1. Step 2 first, because QC is the current blocking full-run failure.
2. Step 3 second, because metrics and downstream outputs create the next largest
   tables.
3. Step 1 preprocessing Slurm wrapper and chunked record coverage.
4. Steps 4-6 partitioned loaders and chunked spatial/GeoJSON helpers.
5. Step 7 dashboard preparation and dashboard lazy-loading.

## Validation Strategy

- Unit tests for each chunked helper using small CSV/Parquet inputs with forced
  small chunk sizes.
- CLI tests for Slurm script generation without submitting jobs.
- Notebook JSON validation for all edited notebooks.
- Do not execute full notebooks locally; validate CARC-oriented commands on the
  cluster when needed.
